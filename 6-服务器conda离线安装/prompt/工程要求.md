# 项目设计说明

你要扮演一名资深 DevOps 工程师与 Python 工程师，为一个有外网的本地开发机和一个无法访问互联网的内网 Linux 服务器设计并实现一套长期稳定维护的 Conda 环境离线分发系统。

## 业务背景

我有如下场景：

- 本地机器可以联网
- 内网服务器不能联网
- 两者可以通过人工传输文件
- 目标是让内网服务器尽可能稳定、可重复地获得与本地一致的开发环境
- 我不追求“简单复制整个 env 目录”，而追求长期可维护、可审计、可重复构建
- 主要面向 Linux
- 优先支持 Conda 环境，且环境中可能同时存在 pip 安装的包

## 总体目标

请你设计并实现一个工程化项目，生成一套完整代码与目录结构，实现以下能力：

1. 环境导出

- 从指定 conda 环境（以当前名为`ScikitAllele`的conda环境为例）中导出：

- environment.yml
- conda list --explicit 生成的精确依赖文件
- pip freeze 生成的 pip 依赖文件

环境元数据文件，例如：

- 项目名
- 环境名
- 导出时间
- Python 版本
- Conda 版本
- 平台信息
- 架构信息
- 操作系统信息

2. 离线安装材料准备

在联网机器上自动准备：

- conda 离线包缓存
- pip wheel 缓存
- 清晰的产物目录
- manifest 清单文件
- 校验文件，例如 sha256

3. 传输包生成

把上述内容整理为适合人工传输的交付包，例如：

- 一个标准目录或一个 tar.gz 压缩包
- 包内应包含安装说明和 manifest

4. 内网安装

在内网机器上支持：

- 校验传输包完整性
- 使用本地缓存离线创建 conda 环境
- 离线安装 pip 包
- 生成安装日志
- 安装完成后进行基本健康检查

5. 可维护性

整个工程必须满足：

- 结构清晰
- 模块化
- 易扩展
- 易测试
- 错误处理完善
- 日志完善
- 命令行易用


## 设计原则

请严格遵循以下原则：

A. 长期维护优先


B. 不直接复制整个 env 目录

主方案应是：

导出环境描述

收集 conda 包缓存

收集 pip wheel

在目标机器离线重建

可以把 conda-pack 作为可选附加能力，但不是唯一核心路径。

C. 强调可追溯

每次导出都要有：

manifest

元数据

哈希校验

清晰版本信息

D. 强调健壮性

要考虑：

包下载失败

pip 包没有 wheel 只有源码

平台不一致

Python 版本不一致

conda 渠道差异

缓存不完整

目标机器缺少 conda

用户误操作

重复安装

已有同名环境

E. 强调用户体验

CLI 应尽量清晰
